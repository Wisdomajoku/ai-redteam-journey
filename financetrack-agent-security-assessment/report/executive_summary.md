# Executive Summary

## 1. Assessment Overview

FinanceTrack Ltd commissioned a security assessment of the FinanceTrack Transaction Assistant to evaluate its ability to protect sensitive customer financial data and maintain appropriate authorization boundaries during AI-assisted interactions.

The engagement progressed through architecture analysis, automated AI security testing, targeted adversarial validation, and manual verification. The assessment included:

* Architecture review and reconnaissance
* Automated AI security baseline assessment (Garak)
* Targeted adversarial validation (PyRIT Single-Turn)
* Multi-turn conversational validation (PyRIT Crescendo)
* Manual application-layer verification

This staged approach allowed observations from one phase to guide validation in the next, culminating in the confirmation of application-layer behavior before reporting final findings.

---

## 2. Key Finding

The assessment confirmed a **Critical-Severity Broken Object-Level Authorization (BOLA/IDOR)** vulnerability within the FinanceTrack Transaction Assistant.

Manual verification demonstrated that an authenticated user could influence the AI assistant to generate an unauthorized backend request, resulting in the disclosure of another customer's financial records.

**Severity:** **Critical**

The severity reflects the confirmed ability to retrieve another customer's financial information through unauthorized application-layer execution, resulting in a high-impact confidentiality breach.

---

## 3. How the Issue Works (Plain English)

The following simplified architecture illustrates the security flow that was assessed:

```text
[ Customer Browser ]
          │
          │ Conversational Request
          ▼
[ AI Assistant ]
          │
          │ Generates Tool Request
          ▼
[ Backend Server ]
          │
          │ Database Query
          ▼
[ Customer Financial Data ]
```

The exploitation proceeded through four stages.

### 1. Normal Security Boundary

The assistant contains conversational guardrails instructing it to operate only on the authenticated user's information. When presented with direct requests to access another user's account, these guardrails correctly rejected the request.

### 2. Conversational Manipulation

Rather than issuing a direct unauthorized request, an administrative-style message was introduced into the conversation. This influenced the assistant to generate a tool request that conflicted with the intended authorization boundary.

### 3. Backend Authorization Failure

The backend accepted the AI-generated request and queried another user's records without independently verifying that the authenticated session was authorized to access those records.

Instead of enforcing authorization at the server, the application trusted the parameter generated by the AI assistant.

### 4. Unauthorized Data Disclosure

The backend returned sensitive financial records belonging to another customer, and the assistant displayed the information through the user interface.

This confirmed that the application's authorization controls could be bypassed through AI-generated tool requests.

---

## 4. Business Impact

If exploited in a production environment, this vulnerability could expose FinanceTrack Ltd to significant operational and regulatory risks.

### Unauthorized Disclosure of Customer Data

An attacker could potentially obtain transaction histories, financial records, account information, and other sensitive customer data belonging to different users.

### Regulatory and Compliance Risk

Unauthorized disclosure of financial information may violate applicable privacy and financial regulations, including requirements established by frameworks such as the General Data Protection Regulation (GDPR), the California Consumer Privacy Act (CCPA), and the Gramm-Leach-Bliley Act (GLBA).

### Financial and Reputational Impact

A successful exploitation could result in:

* Unauthorized disclosure of customer financial information.
* Regulatory investigations, penalties, or legal action.
* Increased incident response and remediation costs.
* Loss of customer confidence and reputational damage.

---

## 5. Root Cause

The assessment determined that the primary weakness was architectural rather than purely model-related.

The application relied on conversational instructions generated by the AI assistant to influence backend operations without independently enforcing authorization controls.

Although the AI assistant contained conversational guardrails, these controls alone were insufficient because the backend trusted AI-generated parameters when executing sensitive operations.

Language models can assist application workflows, but they should never function as the primary authorization mechanism.

---

## 6. Recommendations

The following actions are recommended in priority order.

### 1. Enforce Server-Side Authorization (Highest Priority)

Every request to retrieve customer information should be validated against the authenticated server-side session before any database query is executed.

The backend must never trust a user identifier supplied directly or indirectly by an AI model.

### 2. Treat the AI Assistant as an Untrusted Component

Authorization decisions must remain the responsibility of backend application logic rather than the language model.

Every AI-generated tool invocation should be validated by server-side authorization controls before interacting with sensitive resources.

### 3. Separate Administrative Functions from User Conversations

Administrative operations such as diagnostics, maintenance tasks, synchronization procedures, and system recovery directives should never be processed through the public chat interface.

Administrative functionality should instead be exposed only through authenticated administrative interfaces protected by appropriate access controls.

### 4. Implement Defense-in-Depth

Authorization should be enforced independently at multiple layers, including:

* Authentication
* Session validation
* Backend authorization
* Database access controls
* Tool invocation validation

Security should not rely solely on conversational guardrails.

---

## 7. Conclusion

This engagement reinforces a fundamental security principle for AI-enabled applications:

> **Large Language Models should assist business logic—not enforce security boundaries.**

The assessment progressed from automated model-layer analysis to confirmed application-layer exploitation using a structured, evidence-based methodology. Automated testing identified potential security signals, targeted adversarial testing validated those behaviors, and manual verification confirmed that the observed model behavior resulted in unauthorized access to sensitive customer data.

Authorization must be enforced by backend application code, independent of any instructions or parameters generated by the language model.
